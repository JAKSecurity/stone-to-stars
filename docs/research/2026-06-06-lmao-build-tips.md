# Build tips extracted from the "LMAO / Temu League of Legends" post

**Source:** r/ClaudeAI post by *jonnygravity* — "I had Opus 4.8 build Temu League of
Legends in under a day" (game: LMAO, lmaomoba.com). Raw copy saved at repo root
(`i_had_opus_48_build_temu_league_of_legends.txt`, gitignored). Author is a 25-year
software engineer; this was a deliberately "vibe-coded" weekend experiment, not his
normal process.

His stack: TypeScript + Canvas (wrapped in React) + PartyKit for multiplayer, ~20k LOC,
mostly hand-rolled, no game engine. (Ours is TypeScript + Phaser + Vite, single-player —
so the multiplayer/PartyKit parts don't apply.)

---

## The actually-useful tips (verbatim-ish), and how they map to Rogue · Civ

### 1. Art is the big unlock: Claude *writes* the art as SVG → procedural Canvas
- "The art is ALL Claude being a masterful painter with SVG." SVGs are "just geometry and
  math," so the model can author them directly. He later had it **convert SVG → procedural
  Canvas draw code** — same technique, but Canvas "unlocks animation and performance."
- Evolution of his art: started as "little pills with emojis for heads" → a pass turning
  them into real SVG-drawn characters → then a "HOLY SHIT mode" pass that built a
  **motion-capable character rig** and a **dedicated sub-agent per character**.
- **Lesson he'd change:** build the **detailed character rig / art foundation FIRST** next
  time — he "burned 2–3x as many tokens" redesigning/animating because he didn't.
- SFX/music were also procedurally generated by Claude (Web Audio). Icons were from
  game-icons.net (CC-BY) — i.e., not literally everything was from scratch.

**→ Rogue · Civ application (highest value):** replace our placeholder colored shapes with
**Claude-authored procedural-Canvas (or SVG) art** for units, enemies, resource gems, and
especially the **base-camp buildings** (the "watch your town grow" payoff deserves real art).
Phaser can render this (draw to a Phaser texture / use generated textures, or draw via a
custom render function). This directly fills our deferred "art pass" and is a reason to
revisit our earlier *pixel-art-via-free-packs* decision — Claude-authored vector/canvas art
is custom, on-theme, and needs no asset hunting. **Do the art-system + one canonical unit's
"rig" first**, ratify the style, *then* fan out per-unit sub-agents (this is exactly our
`calibrate-then-parallelize` skill).

### 2. Sub-agent per focused, singular unit
- One sub-agent per character / ability / asset. "Giving an agent a LOT of leeway to focus
  on a very focused, singular task can yield pretty impressive results." He explicitly
  instructed delegation rather than relying on Claude to choose.

**→ Application:** for P2 content, one sub-agent per **age / tech-cluster / building / enemy
type / unit** (data + its generated art). We already use subagent-driven development; this
says lean into it harder for content, with a calibrated template.

### 3. `/goal` as an autonomous, self-verifying work queue
- First prompt was a big **`/goal`** giving full autonomy: "build the temu version of LoL …
  full autonomy … don't ever stop for my input until you're done … be creative, you're an
  expert game designer/dev." That one-shot a playable POC; ~4–5 such `/goal` passes got it
  "pretty good."
- Then he'd **play, note 10–15 tweaks/bugfixes at a time, and fire them as one `/goal`
  numbered list.** Key mechanic: `/goal`'s stop hook spawns an **evaluator agent** that
  checks whether the goal was actually achieved → much more diligent self-testing, and
  AFK-friendly. Phrase the goal so the evaluator can verify completion.

**→ Application:** our analog is the **playtest-driven punch-list loop** — Jeff plays, lists
N concrete tweaks/bugs, and I execute them as a batch with verification between. (If `/goal`
is available in this Claude Code, use it; otherwise I batch + verify, which is the same
shape.)

### 4. Ultracode Workflows for broad, ambiguous, cross-cutting tasks
- He used **Ultracode Workflows** for things like "optimize performance" and "balance" —
  it plans a phased approach and spins up **100–400 sub-agents**. "Amazing for broad,
  ambiguous tasks." Effort almost always `xhigh` or `ultracode`, thinking on.

**→ Application:** this maps to our **Workflow tool** (opt-in). Good candidates for us: a
**combat/balance pass**, a **bug-hunt sweep**, or a **performance pass** once the game has
more content. Use a workflow when the task is broad and parallelizable, not for a single
targeted fix.

### 5. Prompting style for quality/juice
- "You are in 'HOLY SHIT, AI BUILT THIS!?' mode", "go HOLY SHIT mode on the animations",
  "take this to the next level", **increasingly ambitious** follow-ups, and plain
  **"make it better"** — Claude "is *really* good at iteratively improving its own work"
  with minimal guidance.

**→ Application:** use this push-for-ambition framing specifically in the **art + juice
pass** (particles, screen shake, hit-flash, animated buildings).

### 6. His own caveat — for a *serious* product, spec-drive it
- "If I was gonna build a game more seriously, I'd go through a full, comprehensive planning
  phase first … requirements, architecture, epics, stories (I use BMad)." This build was
  "glorified vibe-coding."

**→ Validation for us:** we already did brainstorm → spec → plan → subagent execution (the
"serious" path he recommends). So the move is to **graft his vibe-coding superpowers — the
SVG art generation, heavier per-unit sub-agent fan-out, Workflows for broad passes, and the
playtest punch-list loop — on top of our spec-driven base.** Best of both.
